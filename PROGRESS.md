## Log

(most recent first)

- Box select via Shift+drag marquee -- the last item in IMPROVEMENTS.md. Turned out react-flow already supports the drag-box gesture out of the box (holding its default `selectionKeyCode` of Shift disables panning and turns a drag into a selection box, confirmed by reading react-flow's own source rather than assuming), so no prop changes were needed for the gesture itself -- the actual gap was that nothing synced the result into App's selection state. Wired via `onSelectionEnd` (fires only when a drag-box gesture completes) rather than the more general `onSelectionChange` (fires on every selection change including ordinary clicks, which would have doubled up with and risked fighting the existing click/ctrl-click selection path -- this distinction mattered enough to trace through react-flow's internals before picking one). Added "Shift+Drag: Box-select multiple nodes" to the shortcuts overlay. Verified live: a plain drag still pans (0 nodes selected, doesn't hijack the default interaction), Shift+drag over four nodes selects exactly those four, the multi-selection panel correctly shows "4 Nodes Selected" with the right rollup stats, and clicking Exclude All on that box-selected set works -- proving it's genuinely wired into the same selection state as every other multi-select feature (bulk exclude/include/delete), not a separate visual-only mechanism.
- Keyboard and ARIA pass on the editing surface: attribute values and the delete-picker's reconnect options were plain clickable divs, unreachable by keyboard and invisible to screen readers as interactive controls -- both are now real `<button>` elements (with `all: unset` plus targeted re-layout for the attribute value, since the global button reset forces uppercase/border/padding that's wrong for arbitrary attribute values), and Enter now opens editing the same as a click. The filter input gained full combobox ARIA (`role="combobox"`, `aria-expanded`, `aria-controls`, `aria-autocomplete="list"`, `aria-activedescendant` tracking arrow-key navigation across the existing `role="listbox"`/`role="option"` structure). Canvas nodes get a real `aria-label` ("Conv, 20 params", "Input x, shape [1, 1, 8, 8]") instead of none, and a `:focus-visible` outline for keyboard users tabbing through the canvas -- had to fight react-flow's own stylesheet, which ships `.react-flow__node.selectable:focus-visible { outline: none }` at higher specificity than a bare `.react-flow__node:focus-visible`, silently eating the first version of this rule (caught via computed-style inspection, not by eye). Verified live: aria-labels read correctly for every node, the focus outline computes to the intended amber/1px only after matching that specificity, the attribute-value button is keyboard-focusable and Enter opens its editor, and the full combobox ARIA chain (expanded/controls/activedescendant) updates correctly as you type and arrow through results.
- StatsBar now uses a CSS container query (not a window-width media query, since the resizable inspector panel means the bar's own width isn't the viewport width) to collapse gracefully instead of cropping content off-screen -- the original audit measured this broken at 1024px: model name clipped off the left, Export Modified/Load new cropped off the right, no wrap or scroll. Below 1280px (container width) the stat labels drop their unit words ("2.4M" instead of "2.4M PARAMS", full label still on hover via title); below 1100px, Benchmark/Download Original/Load New collapse into a single "..." overflow menu (reusing the existing dropdown visual pattern) while Export Modified (the primary action) and the filter stay put. The model name itself also switched from a hard 240px max-width to a flexible `flex: 1 1 120px` that only truncates when genuinely out of room (was already flagged as a gap back in the MiniMap/node-count-grammar entry above, deferred here on purpose). Rendered each stat as two complete parallel text variants (full/compact) rather than nesting the unit word in a conditionally-hidden child span -- the latter reads identically but silently breaks plain-string test queries, since testing-library's default text matcher doesn't concatenate text split across element boundaries (caught this via 4 real test failures, fixed by restructuring rather than rewriting the tests around the wrong architecture). Verified live at 1440/1024/900px: at 1024px the model name stays fully visible, stats compact correctly, and the overflow menu opens with all three collapsed actions intact.
- A plain click on any edge used to insert a passthrough node immediately and silently (verified live in the original audit: node count 3 to 4 on a single unlabeled click). Clicking an edge now opens a small popover at the click point with the tensor name and an "Insert passthrough" button; the insert only happens if that button is clicked. Escape or clicking anywhere else (a node, the pane) dismisses without side effects. The selected edge also thickens while its popover is open. A successful insert now announces "Inserted passthrough on X (Ctrl+Z to undo)" via the annunciator line; an ineligible target announces why instead of silently doing nothing. Not testable in jsdom (react-flow never populates edge DOM there -- confirmed by dumping it directly, and no prior test in this codebase exercised edge clicks for the same reason), so verified fully live instead: plain click leaves node count unchanged and shows the popover, Escape and pane-click both dismiss without inserting, and only the confirm button increases the node count and fires the announcement.
- Restructured the Layer Inspector's node-detail panel: merged the two sections that both listed every tensor name ("Input shapes" and "Inputs", same for outputs) into one row per tensor with the shape shown alongside the name when known, instead of twice. Reordered to identity -> attributes -> inputs/outputs -> stats -> a visually separated action zone (a divider, then exclude/delete) instead of interleaving destructive actions between stats and content. SENSITIVITY keeps its label but gets a title tooltip clarifying it's a param-count heuristic, not a real measurement. EXCLUDED's YES/NO toggle became a self-labeled action button ("Exclude from stats" / "Include in stats") with a tooltip explaining it only affects rollups/dimming and never export. All 261 existing tests passed unmodified (none asserted on the old duplicated sections). Verified live (screenshot).
- CSS-only motion system (no framer-motion/react-spring -- their one edge, exit animations, isn't needed here since the glide rule below makes deletions legible on its own): a 180ms transition on `.react-flow__node` transform so delete/insert/rewire/layout-toggle glide surviving nodes to their new dagre position instead of snapping (disabled during an active manual drag, where it would just lag the cursor); a subtle opacity-in on node mount; a marching-ants dash animation on the active drag-to-rewire connection line; an amber pulse on a valid drop-target handle; a shimmer sweep on the loading progress bar. All gated behind `prefers-reduced-motion: no-preference`. Verified live: computed style confirms the transition and animation are actually applied (not just written and unused), and a real drag-to-rewire shows the dashed marching-ants line mid-drag (screenshot).
- A model can now be dropped onto the app at any time to replace the one currently open, not just from the initial empty state -- previously the only way back to a file picker was the "Load new" button. Dragging a file over the loaded workspace shows a full-viewport "Drop to replace current model" overlay (with an "Unsaved edits will be lost" warning when there are pending edits), guarded by a depth counter so dragenter/dragleave flicker between child elements doesn't flash the overlay on and off. Verified in a real browser (not just jsdom) by dispatching native DragEvents: overlay appears on dragenter, the model actually swaps on drop (node count went 3 to 8, confirming tiny.onnx was replaced by the sample model), overlay clears on dragleave without loading anything.
- Implemented together since one needs the other: worker ERROR messages now carry a `scope: 'load' | 'operation'` tag (LOAD_MODEL failures vs everything else). Load errors keep the existing full-screen dropzone/error behavior; operation errors (a failed benchmark, export, or inference) now keep the loaded graph mounted and status at 'ready' instead of tearing the whole workspace down for what might just be a transient WASM hiccup. A new always-present 22px status strip under the StatsBar (an "annunciator line," not a floating toast, per the audit's explicit design note) surfaces operation errors plus several previously-silent actions: rejected rewires (with the real reason -- self-connect, cycle, wrong node type), bulk-delete skip counts when nodes were ambiguous/ineligible, undo confirmations naming what was undone, and a copy confirmation. Verified live: self-connecting a node's output to its own input shows "Rewire rejected: Cannot connect a node to itself", Ctrl+Z after an attribute edit shows "Undid kernel_shape edit", and COPY shows "Copied to clipboard" -- all in the strip, with the graph staying fully interactive throughout.

- The single highest-impact fix in the whole list: `onnxWorker.ts` used to post `MODEL_LOADED` only after `ort.InferenceSession.create` resolved, so first paint of the graph waited on a full WASM runtime download+compile even though the graph itself was already parsed. The original live-site audit measured this at roughly 60 seconds over a real network. Session creation is now lazy (`ensureSession()`), triggered only by the first Benchmark click or inference call, from the same retained `exportBuffer` bytes. Verified live: graph now renders in ~120ms after picking a file (session doesn't exist yet), Benchmark still works correctly on first click (lazy creation succeeds, warmup+median intact), and switching to a second model correctly resets and recreates the session rather than reusing a stale one (added an explicit `session = null` reset in the LOAD_MODEL handler, since the module-level session variable used to only ever get reassigned, never cleared).
